# Requirements: Add Web Serve Structured Logging

## 1. Background

`web serve` starts a NestJS application through `src/web/main.ts`. The bootstrap
currently disables Nest's logger (`logger: false`), so server startup, requests,
and server-side failures have no structured operational record. Web routes include
album and audiobook HTTP APIs plus the `/mcp` Streamable HTTP endpoint.

The 2026-07-18 `add-web-serve-supercommand` spec established the current bootstrap
and HTTP behavior. This follow-up adds Pino logging only to the web-server process;
the Commander CLI's existing result output remains separate.

## 2. Goal

`web serve` MUST emit machine-readable JSON logs through Pino for server lifecycle,
completed HTTP requests, and unexpected request failures, without changing routes,
response shapes, status codes, MCP protocol responses, or the dry-run safeguards of
the existing operations.

## 3. Scope

### In scope

- Add the `pino` runtime dependency in `package.json` and update `package-lock.json`
  through npm.
- Add a focused logger adapter and request/error logging integration under
  `src/web/**`.
- Wire the logger into `src/web/main.ts` and `serveWeb`.
- Add or update focused tests in `__tests__/web/**`.
- Add a concise `web serve` logging section to the existing user-facing web/MCP
  documentation if a relevant document is present.

### Out of scope

- Logging for non-web Commander commands or library/media operations.
- Browser UI, authentication, persistence, telemetry exporters, log shipping, or
  log rotation.
- Request-body logging, audio metadata logging, or logging filesystem paths passed
  by clients.
- Changing endpoint paths, request schemas, response payloads, error envelopes,
  MCP transport behavior, source/destination confinement, or write semantics.
- Adding `pino-http`, `nestjs-pino`, or another logging framework unless a directly
  blocking compatibility issue requires an approved scope revision.

## 4. Functional Requirements

- **FR-1** `package.json` MUST declare `pino` as a runtime dependency, and
  `package-lock.json` MUST be regenerated by npm to match.
- **FR-2** `web serve` MUST write newline-delimited JSON logs to stderr, preserving
  stdout for CLI output and preventing log lines from corrupting MCP responses.
- **FR-3** The web bootstrap MUST emit an `info` lifecycle record after the HTTP
  listener is ready, including a stable event name and the bound host and port.
- **FR-4** The web application MUST emit exactly one completion record per HTTP
  request after the response finishes, including a stable event name, request
  method, URL path without query parameters, response status code, duration in
  milliseconds, and a request ID.
- **FR-5** Completion records MUST classify HTTP 5xx responses at `error` level and
  all other completed responses at `info` level; expected 4xx client failures MUST
  NOT be logged as errors.
- **FR-6** Unexpected request failures that reach the web exception boundary MUST
  emit an `error` record with Pino's standard serialized error field and the
  request ID before the existing HTTP response behavior proceeds.
- **FR-7** Incoming `x-request-id` values MAY be reused only after validating them
  as bounded printable identifiers; otherwise the server MUST generate an opaque
  request ID and return it in the response `x-request-id` header.
- **FR-8** Logs MUST NOT include request bodies, authorization/cookie headers,
  source or destination root paths, individual file paths, raw query strings, or
  media metadata.
- **FR-9** The Pino integration MUST preserve current Nest controller dispatch,
  HTTP status codes and bodies, `/mcp` response streaming, and CLI output behavior.
- **FR-10** Logger construction and the request/error integration MUST allow a
  deterministic in-memory destination in tests without opening a network listener
  or inspecting process-global stdout/stderr.

## 5. Non-Functional Requirements

- **NFR-1 — Lint after every source code file modification** After every
  modification of a source code file, `npm run lint -- <modified-file>` MUST be run
  so only the modified file is linted, and any reported issues MUST be fixed before
  moving on. This applies per source-code edit, not per task. Whole-codebase
  `npm run lint` MUST be reserved for final verification after all TypeScript
  modifications are complete.
- **NFR-2 (build)** `npm run build` MUST exit 0 after implementation.
- **NFR-3 (tests)** `npm test` MUST exit 0 after implementation.
- **NFR-4 — No `npx`** `npx` is forbidden in all forms. Commands MUST use
  `npm run <script>` or `./node_modules/.bin/<tool>` exclusively.
- **NFR-5 (type safety)** The implementation MUST use strict TypeScript without
  `any` or TypeScript suppression directives.
- **NFR-6 (file size)** No source or test file created by this spec MAY exceed
  200 lines.
- **NFR-7 (scope discipline)** `git --no-pager diff --stat -- src/commands
  src/lib bin etc reports` MUST be empty after implementation.
- **NFR-8 (behavioral parity)** Existing route contracts, controller tests, MCP
  protocol behavior, and dry-run/execute semantics MUST remain unchanged.

## 6. Acceptance Criteria

1. A started `web serve` process emits parseable Pino JSON lifecycle records on
   stderr and does not emit those records on stdout.
2. Focused tests prove one completion record with the required safe fields for a
   successful request, a 4xx request, and a 5xx request.
3. Focused tests prove request IDs are returned and correlated with completion and
   unexpected-error records, without logging prohibited sensitive fields.
4. `npm run lint`, `npm run build`, and `npm test` exit 0.
5. The final diff contains only the files enumerated in `design.md` section 2.
